# 《AI 面试嘴替》技术方案 PDF 修改清单

> 适用对象：产品、方案撰写、路演材料制作、开发验收  
> 评估对象：`AI面试嘴替.pdf`，共 10 页  
> 基线日期：2026-07-09  
> 当前实现基线：Next.js App Router + React + TypeScript + OpenAI-compatible LLM + Azure TTS/STT + Web Speech/文本兜底

## 1. 修改目标

本次不推翻 PDF 的产品叙事。需要完成三类修正：

1. 把已经实现的能力描述准确，避免方案稿低估当前实现。
2. 删除尚未实现或未经实测的技术承诺，避免验收时形成错误预期。
3. 统一 PDF、页面、Prompt 和项目契约中的面试官名称与能力边界。

## 2. 修改优先级总览

| Priority | 必须修改项 | 原因 |
| --- | --- | --- |
| P0 | 统一三种面试官名称和人设 | PDF、代码展示、`AGENTS.md` 当前存在三套口径 |
| P0 | 修正“真实流式”和 TTFT `< 1.5 秒` | 当前是 SSE 进度流，模型仍一次性返回完整 JSON |
| P0 | 修正 SPA/Vercel 部署描述 | 当前正式实现是 Next.js 全栈，推荐腾讯云 Node 自托管 |
| P0 | 删除“100% 稳定”等绝对承诺 | 无法作为可验证工程指标 |
| P1 | 修正语音链路描述 | 当前是 Azure TTS/STT 优先，Web Speech 和文本兜底 |
| P1 | 修正“题型权重”描述 | 当前没有 `weight` 字段，只有 Prompt 规则和 `expectedSignals` |
| P1 | 修正高分答案示例中的虚构数据 | 30%、45%、3 人团队等数据不能脱离用户材料生成 |
| P2 | 收敛 Agent、壁垒和长期生态表述 | 当前产品是受控工作流，不是自治 Agent 或就业基础设施 |

#### 建议替换

### 第 4 页：5 分钟流程

#### 0-1 分钟

建议描述：

> 输入或上传简历与目标 JD。  
> 系统生成候选人画像、岗位匹配点、风险点与证据来源。

#### 1-3 分钟

当前描述：

> 基于题型权重生成 3 道高命中面试题。  
> 前端 TTS 读题，Web Speech 语音直答。

建议替换：

> 基于候选人画像、JD 匹配证据和所选面试官人设生成 3 道针对性问题。  
> Azure TTS 服务端合成优先，失败时降级为 Web Speech 或题目文本。  
> 答题优先使用 Azure STT，失败时可重试、保留已识别文本或手动编辑。

#### 3-5 分钟

当前描述：

> 流式传输极速响应。输出多维雷区诊断与高分嘴替改写台词。

建议替换：

> 页面通过 SSE 即时展示生成进度，完成后输出六维评分、风险标签、致命问题、优化答案和 60 秒口述版。  
> 流式失败时保留问题和答案，并自动尝试非流式报告或演示兜底。

#### 时间口径补充

页面上的 `0-1 / 1-3 / 3-5 min` 应标注为：

> 标准 Demo 演示节奏，不作为接口性能 SLA。

#### 是否改代码

否。若要把 5 分钟定义为正式 SLA，需要另加端到端计时埋点和线上验收。

#### 当前标题

#### 建议标题

> 出题覆盖原则：岗位匹配 + 简历证据 + 高频场景 + 面试官人设

#### 建议补充说明

> 当前三项依据属于 Prompt 出题规则，不对外展示数值权重。每题通过 `intent` 和 `expectedSignals` 保留出题意图与评分依据。

#### 三种面试官统一文案

以 `AGENTS.md` 硬边界为准，PDF 建议统一为：

| 面试官 | 建议描述 |
| --- | --- |
| 大厂严厉 HR | 关注岗位匹配、求职动机、稳定性和经历真实性，直接追问风险点与短板 |
| 技术老哥 | 关注项目深度、方法论、技术或产品边界、异常处理和个人贡献 |
| 温柔大姐姐 | 关注经历梳理、表达结构和成长潜力，使用引导式追问帮助候选人说清楚 |

原页面中 Q1/Q2/Q3 分别绑定三位面试官的表达建议删除，替换为：

> 用户先选择一位面试官；随后 3 道题均受该人设影响，同时覆盖经历、岗位匹配、专业深挖或压力场景。

#### 修改原因

- 当前产品不是“三道题分别由三种面试官提问”，而是“选择一种风格后完成三题”。
- 当前 schema 没有题目 `weight`，不应宣传为可计算的严格权重模型。
- 当前代码展示名称仍是“温柔HR小姐姐 / 技术老哥 / 资深业务大佬”，需要同步修改，见第 5 节。

#### 当前问题

如果这些信息不来自用户简历或回答，就违反当前事实边界。

#### 建议替换示例

> [T] 我负责【请补充本人职责】，目标是在【请补充真实周期】内解决核心问题。  
> [A] 我先定位原因，再采取【请补充真实动作】，并与【请补充真实协作对象】推进落地。  
> [R] 最终取得【请补充真实结果】。如果暂无量化指标，可以说明交付物、用户反馈或业务决策结果。

底部边界说明建议改为：

> 事实边界：优化答案只重组用户已提供的信息；缺少人数、周期、金额或指标时使用明确占位符，不自动补造数据。

按钮文案保留：

### 第 8 页：技术架构与性能承诺

#### 当前“极轻量前端部署”

> SPA 单页应用架构。Vercel 一键极简部署，即开即用。

#### 建议替换

> Next.js App Router 全栈应用。前端、Route Handlers、LLM 代理、Azure TTS/STT 代理和 SSE 报告接口使用同一代码仓库。  
> 当前推荐腾讯云 Node.js + PM2 + Nginx 自托管，便于保存全局 Prompt 并控制调试入口。Vercel 可作为备选，但需先替换本地文件持久化方案。

#### 当前“性能承诺”

> 大模型 Stream 流式传输。核心指标：首字响应时间 TTFT < 1.5 秒。

#### 建议替换

> 报告接口使用 SSE 立即反馈排队、评分和汇总进度；最终事件返回完整结构化报告。  
> 当前模型仍一次性生成完整 JSON，不承诺模型 token 级 TTFT。报告生成超时上限为 60 秒，失败时切换非流式或演示兜底。

如确实需要性能指标，建议改为可测口径：

| Metric | 建议目标 | 前置条件 |
| --- | --- | --- |
| SSE 建连并显示首个进度 | `< 1.5s` | 增加服务端和前端埋点 |
| 完整报告成功率 | 现场演示前实测并记录 | 固定模型、网络和 Prompt |
| 完整报告耗时 | P50/P95 实测后填写 | 至少 30 次真实请求 |
| 降级后闭环完成率 | 100% 通过验收脚本 | 使用内置演示样例包 |

#### 当前“100% 稳健交付”

建议替换：

> 多级降级保证主流程可继续：Azure TTS/STT -> Web Speech -> 文本；真实 LLM -> 重试 -> 非流式 -> 内置演示样例。

#### 是否改代码

- 当前无需为改文案修改代码。
- 若要承诺真实流式，需要改造 LLM provider 为流式输出，并解决结构化 JSON 增量校验问题。
- 若要承诺性能指标，需要新增埋点、压测和发布后监控。
